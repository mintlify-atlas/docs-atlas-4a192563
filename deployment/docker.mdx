## Overview

Enigma Developers uses Docker Compose to orchestrate a multi-container application with three core services: frontend (React), backend (Express), and database (PostgreSQL).

## Architecture

The application consists of the following services:

- **Frontend**: React application served via Nginx
- **Backend**: Express.js API server with Prisma ORM
- **Database**: PostgreSQL 16 (Alpine-based)
- **Network**: Bridge network (`enigmasac-net`) for inter-service communication

## Docker Compose Configuration

### Development Setup

The main `docker-compose.yml` file is configured for development environments:

```yaml docker-compose.yml
services:
  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    container_name: enigmasac-frontend
    restart: unless-stopped
    ports:
      - "4051:80"
    networks:
      - enigmasac-net

  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    container_name: enigmasac-backend
    restart: unless-stopped
    env_file: .env
    ports:
      - "4050:4000"
    depends_on:
      db:
        condition: service_healthy
    networks:
      - enigmasac-net

  db:
    image: postgres:16-alpine
    container_name: enigmasac-db
    restart: unless-stopped
    env_file: .env
    environment:
      - POSTGRES_DB=${POSTGRES_DB}
      - POSTGRES_USER=${POSTGRES_USER}
      - POSTGRES_PASSWORD=${POSTGRES_PASSWORD}
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER} -d ${POSTGRES_DB}"]
      interval: 5s
      timeout: 5s
      retries: 5
    networks:
      - enigmasac-net

volumes:
  pgdata:

networks:
  enigmasac-net:
    driver: bridge
```

<Note>
  The backend service uses `depends_on` with `condition: service_healthy` to ensure the database is fully ready before starting.
</Note>

## Service Dockerfiles

### Frontend Dockerfile

The frontend uses a multi-stage build to optimize the final image size:

```dockerfile frontend/Dockerfile
FROM node:20-alpine AS build
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm install
COPY . .
RUN npm run build

FROM nginx:alpine
COPY --from=build /app/dist /usr/share/nginx/html
RUN chmod -R a+r /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

**Build stages:**
1. **Build stage**: Installs dependencies and builds the React application
2. **Production stage**: Serves the built files using Nginx Alpine

### Backend Dockerfile

The backend uses a production-optimized Node.js image:

```dockerfile backend/Dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package.json ./
RUN npm install --omit=dev
COPY . .
RUN npx prisma generate
EXPOSE 4000
CMD ["npm", "start"]
```

<Note>
  The backend installs only production dependencies (`--omit=dev`) and generates Prisma client during the build.
</Note>

## Frontend Nginx Configuration

The frontend nginx configuration handles static file serving, caching, and language-specific routing:

```nginx frontend/nginx.conf
server {
    listen 80;
    server_name localhost;
    root /usr/share/nginx/html;
    index index.html;

    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml image/svg+xml;
    gzip_min_length 256;

    location /assets/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    location ~ ^/en(/.*)?$ {
        try_files /index.html =404;
        sub_filter_once off;
        sub_filter_types text/html;
        sub_filter '<html lang="es"' '<html lang="en"';
        # ... additional substitutions for English version
    }

    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

**Key features:**
- Gzip compression for text-based assets
- 1-year caching for static assets
- Dynamic language substitution for English routes
- SPA fallback to index.html

## Health Checks

### Database Health Check

The PostgreSQL service includes a health check that ensures the database is ready:

```yaml
healthcheck:
  test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER} -d ${POSTGRES_DB}"]
  interval: 5s
  timeout: 5s
  retries: 5
```

This health check:
- Runs every 5 seconds
- Times out after 5 seconds
- Retries up to 5 times before marking as unhealthy
- Uses `pg_isready` to verify PostgreSQL is accepting connections

## Docker Commands

<Steps>
  <Step title="Start all services">
    Start the entire stack in detached mode:
    
    ```bash
    docker-compose up -d
    ```
  </Step>

  <Step title="View logs">
    Monitor logs from all services:
    
    ```bash
    docker-compose logs -f
    ```
    
    Or view logs for a specific service:
    
    ```bash
    docker-compose logs -f backend
    ```
  </Step>

  <Step title="Stop services">
    Stop all running containers:
    
    ```bash
    docker-compose down
    ```
    
    Stop and remove volumes (warning: deletes data):
    
    ```bash
    docker-compose down -v
    ```
  </Step>

  <Step title="Rebuild services">
    Rebuild containers after code changes:
    
    ```bash
    docker-compose up -d --build
    ```
  </Step>

  <Step title="Execute commands in containers">
    Run database migrations:
    
    ```bash
    docker-compose exec backend npm run db:migrate
    ```
    
    Access PostgreSQL shell:
    
    ```bash
    docker-compose exec db psql -U enigma_user -d enigma
    ```
  </Step>
</Steps>

## Service Dependencies

The service startup order is managed through dependencies:

1. **Database** starts first and runs health checks
2. **Backend** waits for database to be healthy before starting
3. **Frontend** starts independently (no backend dependency)

<Warning>
  Always ensure the database is healthy before running migrations or connecting from the backend.
</Warning>

## Port Mappings

| Service  | Container Port | Host Port | Description                    |
|----------|----------------|-----------|--------------------------------|
| Frontend | 80             | 4051      | Nginx serving React app        |
| Backend  | 4000           | 4050      | Express API server             |
| Database | 5432           | N/A       | PostgreSQL (internal only)     |

<Note>
  In development, ports are exposed on all interfaces. See the [Production Guide](/deployment/production) for secure port binding.
</Note>

## Volumes

### Database Volume

The `pgdata` volume persists PostgreSQL data:

```yaml
volumes:
  pgdata:
```

This ensures database data survives container restarts and rebuilds.

## Networking

All services communicate through the `enigmasac-net` bridge network:

```yaml
networks:
  enigmasac-net:
    driver: bridge
```

**Internal hostnames:**
- `frontend` - React application container
- `backend` - Express API container
- `db` - PostgreSQL container

Services reference each other using these hostnames (e.g., `DATABASE_URL=postgresql://user:pass@db:5432/enigma`).

## Troubleshooting

### Backend won't start

Check if the database is healthy:

```bash
docker-compose ps
```

If the database is unhealthy, check logs:

```bash
docker-compose logs db
```

### Port conflicts

If ports 4050 or 4051 are already in use, modify the port mappings in `docker-compose.yml`:

```yaml
ports:
  - "4052:80"  # Change host port to 4052
```

### Database connection errors

Verify environment variables are correctly set in `.env` file. See [Environment Variables](/deployment/environment) for details.
